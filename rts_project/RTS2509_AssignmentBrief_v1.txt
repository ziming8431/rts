Precision Control in an Automated Manufacturing Line: Rust-Based Sensor-Actuator
 
In a modern automated manufacturing plant, robotic arms perform high-speed assembly, welding, and packaging with extreme precision. These systems rely on real-time sensor feedback to dynamically adjust their movements — applying the right pressure, compensating for vibration, or realigning components on a conveyor belt. Even slight response delays can lead to misalignment, defective products, or costly downtime.
Your task is to develop a 
complete real-time sensor–actuator system
 using Rust. One part of your system will 
simulate and process sensor data
, while the other will 
control the robotic arm
 based on this data, maintaining smooth and timely responses. Both modules must operate within strict real-time constraints and communicate efficiently.
Core System Requirements
Component A – The Sensor Data Simulator
Generate Sensor Data
Simulate sensors (e.g., force, position, temperature) providing readings at fixed intervals (e.g., every 5 ms).
Maintain consistent sampling rates using real-time threads or async constructs.
Process Data
Apply noise-reduction filters (e.g., moving average).
Implement anomaly detection.
Ensure each processing cycle completes within 0.2 ms (simulated constraint).
Shared Resource Synchronisation 
Both modules must periodically access a 
shared resource
 (e.g., diagnostic log, configuration buffer).
Implement appropriate synchronisation using mutexes, atomics, or lock-free structures.
Benchmark and discuss lock contention and priority inversion effects.
Transmit Data in Real Time
Send processed data to the actuator module using an efficient IPC mechanism (e.g., channel, shared memory).
Data must be transmitted within 0.1 ms after processing (simulated deadline).
Benchmark Performance
Record execution times for each stage (generation, processing, transmission).
Measure latency, jitter, and throughput under normal and high-load conditions.
Compare different concurrency constructs where appropriate.
Component B – The Actuator Commander
Receive Sensor Data
Implement a receiver module that handles sensor input efficiently with minimal delay.
Control the Robotic Arm (Predictive Control)
Implement a virtual actuator responding to sensor inputs (e.g., adjusting grip, correcting misalignment).
Integrate a predictive control algorithm (PID or equivalent) to adjust actuation dynamically.
Apply real-time scheduling techniques to prioritise critical tasks.
Manage Multiple Actuators
Extend your design to handle multiple actuators (grippers, motors, stabilisers) concurrently.
Ensure each actuator operates within its assigned deadline (1–2 ms simulated).
Close the Feedback Loop
Send feedback to the sensor module (acknowledgements, actuator state, error data).
Allow dynamic recalibration or threshold adjustment based on feedback.
Feedback must be sent within 0.5 ms of actuation (simulated).
Benchmarking & Analysis
Measure and discuss system performance under varying load conditions.
Log and analyse missed deadlines, throughput, and scalability.
Note on Timing Constraints
T
he specified response and transmission deadlines (e.g., 0.1 ms, 0.5 ms) are 
intentionally tighter than achievable on a standard OS environment
.
They are meant to 
encourage experimentation
 with concurrency constructs, task scheduling, and optimisation strategies.
Students are expected to:
Measure and compare actual performance under different designs.
Identify and explain performance bottlenecks.
Discuss trade-offs between safety, concurrency, and responsiveness in their report.
System Integration Requirements
To demonstrate a complete end-to-end real-time control system, you must 
integrate both the Sensor (Component A)
 and 
Actuator (Component B)
 modules. Integration must enable live communication, feedback, and benchmarking.
Core Integration Options (choose one):
Multi-Threaded Integration 
Implement both modules as concurrent threads within a single Rust program.
Use

,

 channels).
Ensure deterministic timing, handle contention safely, and measure end-to-end latency.
Asynchronous Integration

Implement both modules as asynchronous tasks sharing data via async channels or event streams.
Benchmark and compare the async design against a threaded version where possible.
Your chosen architecture must be clearly documented and justified in your report. End-to-end performance benchmarking must include both sensor and actuator cycles.

Fault Injection / Sensor Dropouts

Implement fault detection, recovery, and error logging mechanisms.
Demonstrate that the system remains stable under transient failures.
Fail-Safe Mode / Graceful Degradation
Define a safe fallback behaviour when timing constraints or sensor data quality thresholds are violated.
Implement logic to enter a reduced-functionality state (e.g., limit actuator motion, pause updates).
Evaluate recovery mechanisms.
CPU Load Simulation
Introduce background non-real-time threads or workloads to simulate system interference.
Measure how increasing load affects real-time performance (latency, missed deadlines).
Present quantitative analysis with visualisations.
Real-Time Visualisation Dashboard

Discuss how message queuing, buffering, and delivery guarantees affect real-time responsiveness and safety.
